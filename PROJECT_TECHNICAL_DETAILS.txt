========== FSRS算法详解 ==========

1. 算法原理

FSRS（Free Spaced Repetition Scheduler）是一种基于遗忘曲线的记忆算法。

核心参数：
- 难度（Difficulty, D）：单词的学习难度，范围0.15-0.98
- 稳定性（Stability, S）：单词的记忆稳定性，单位为天
- 可检索性（Retrievability, R）：能回忆起单词的概率，范围0.05-0.99
- 复习间隔（Interval, I）：下次复习的天数

2. 初始化

新单词初始化：
- difficulty_score = 0.35（中等难度）
- stability = 1.5（1.5天）
- retrievability = 0.92（92%概率回忆）
- interval_days = 0
- lapse_count = 0（失误次数）
- review_count = 0（复习次数）

3. 答对时的状态更新

当用户答对时：

步骤1：计算经过的天数
elapsed_days = (当前时间 - 上次复习时间) / 86400000

步骤2：计算新的可检索性
retrievability = retrievability * exp(-elapsed_days / stability)

步骤3：计算新的难度
difficulty_score = difficulty_score - 0.06 * (1 - retrievability)
difficulty_score = clamp(difficulty_score, 0.15, 0.98)

步骤4：计算新的稳定性
stability = stability * (1 + 8 * (1 - difficulty_score) * (retrievability - 0.5) / (1 - 0.5))
stability = max(stability, 0.2)

步骤5：计算新的复习间隔
interval_days = stability * 0.7 * (1 - 0.9 * difficulty_score)
interval_days = min(interval_days, 90)

步骤6：计算下次复习时间
next_review_time = 当前时间 + interval_days * 86400000

步骤7：更新复习计数
review_count += 1
retrievability = 0.97

4. 答错时的状态更新

当用户答错时：

步骤1：计算经过的天数
elapsed_days = (当前时间 - 上次复习时间) / 86400000

步骤2：计算新的可检索性
retrievability = retrievability * exp(-elapsed_days / stability)

步骤3：计算新的难度
difficulty_score = difficulty_score + 0.12 * (1 - retrievability)
difficulty_score = clamp(difficulty_score, 0.15, 0.98)

步骤4：计算新的稳定性
stability = stability * 0.42

步骤5：计算新的复习间隔
interval_days = 0.04（1小时，立即复习）

步骤6：计算下次复习时间
next_review_time = 当前时间 + interval_days * 86400000

步骤7：更新复习计数
lapse_count += 1
retrievability = 0.35

5. 参数调优

根据用户学习速度调整参数：

快速学习者（learningSpeed = 'fast'）：
- STABILITY_GROWTH_FACTOR = 2.0（加快学习进度）
- INITIAL_STABILITY = 2.0
- INTERVAL_BASE_FACTOR = 0.9

正常学习者（learningSpeed = 'normal'）：
- STABILITY_GROWTH_FACTOR = 1.8
- INITIAL_STABILITY = 1.5
- INTERVAL_BASE_FACTOR = 0.7

慢速学习者（learningSpeed = 'slow'）：
- STABILITY_GROWTH_FACTOR = 1.3（延长复习间隔）
- INITIAL_STABILITY = 1.0
- INTERVAL_BASE_FACTOR = 0.5

========== 虚拟滚动优化 ==========

1. 原理

虚拟滚动（Virtual Scrolling）是一种性能优化技术，只渲染可见区域的元素，而不是渲染整个列表。

优势：
- 减少DOM节点数量
- 降低内存占用
- 提高滚动帧率
- 支持大列表渲染

2. 实现细节

VirtualScroller.vue组件：

属性：
- items：要渲染的数据数组
- itemHeight：每个项目的高度（像素）
- bufferSize：缓冲区大小（项目数）
- containerHeight：容器高度（像素）

计算：
- visibleCount = Math.ceil(containerHeight / itemHeight)
- startIndex = Math.floor(scrollTop / itemHeight) - bufferSize
- endIndex = startIndex + visibleCount + 2 * bufferSize
- offsetY = startIndex * itemHeight

渲染：
- 只渲染 items[startIndex:endIndex] 的项目
- 使用 transform: translateY(offsetY) 定位
- 监听滚动事件，动态更新 startIndex 和 endIndex

3. 性能提升

优化前：
- 渲染1000个项目
- DOM节点数：1000+
- 内存占用：~50MB
- 滚动帧率：30-40 FPS

优化后：
- 渲染1000个项目
- DOM节点数：20-30（可见+缓冲）
- 内存占用：~5MB
- 滚动帧率：60 FPS

========== 多数据源统一查询 ==========

1. 数据源优先级

优先级1：用户库（最高）
- 用户自定义的单词
- 用户导入的单词
- 用户修改过的单词

优先级2：主库（中等）
- vocal_master.db
- 包含预生成词库、真题统计、释义等
- 由build_master_db.py生成

优先级3：预生成库（低）
- pregen_data.db
- 包含预生成的词汇信息
- 作为兜底方案

2. 查询策略

两阶段加载：

第一阶段（轻量首帧）：
- 从用户库查询基本信息（英文、中文、标签）
- 快速返回，提高首屏加载速度
- 时间：< 100ms

第二阶段（异步补全）：
- 从主库查询详细信息（例句、同义词、真题统计）
- 异步加载，不阻塞UI
- 时间：< 500ms

3. 缓存策略

内存缓存：
- 使用LRU缓存
- 最大容量：1000条记录
- 过期时间：5分钟

存储缓存：
- 使用localStorage（H5）或SQLite（App）
- 最大容量：500条记录
- 过期时间：24小时

  "stream": false

  "choices": [

      "message": {

      }
    }
  ]
}
