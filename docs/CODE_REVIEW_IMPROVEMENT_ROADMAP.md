## 🔴 第一阶段：关键改进（性能和稳定性）

### 1. N+1 查询问题 - 数据库性能瓶颈 🔴 高优先级

**问题描述**
- `pages/index/index.vue` 中的 `loadCounts()` 函数对每个单词本逐个调用数据库查询
- 在有 10+ 个单词本时，会产生 10+ 次数据库查询
- `pages/review/review.vue` 中多次调用 `db.getAllWords()` 来获取全量数据

**影响范围**
- 单词本列表页面加载时间 O(n)
- 复习页面初始化时间过长
- 用户体验：明显的加载延迟

**建议的解决方案**
- 实现批量查询接口：`db.getWordsCountBatch(bookIds)` 返回 `{bookId: count}` 映射
- 在 `learningCenter_v2.js` 中缓存单词本统计数据，设置 1 小时过期
- 使用 `getWordsForList()` 的 `LIMIT 1` 替代 `getAllWords()` 来获取计数

**预期收益**
- 单词本列表加载时间从 2-3 秒降至 200-300ms
- 复习页面初始化时间减少 50%
- 数据库查询次数减少 80%

**工作量**：4-6 小时

---

### 2. 内存泄漏风险 - 全量数据加载 🔴 高优先级

**问题描述**
- `pages/index/index.vue` 中的 `allExternalWords = []` 非响应式全量缓存，用于存储 6000+ 词
- 每次页面加载都会重新加载全量数据到内存
- 没有清理机制，页面切换时数据仍保留在内存中
- `learningCenter_v2.js` 中的缓存每次都从存储读取并解析 JSON

**影响范围**
- 内存占用：6000+ 词 × 每词 ~500 字节 = 3MB+ 持续占用
- 长期使用应用会导致内存逐渐增长
- 低端设备（内存 < 2GB）会出现卡顿

**建议的解决方案**
- 实现分页加载：初始加载 200 词，滚动到底部时加载下一页
- 使用 WeakMap 存储大对象引用，允许垃圾回收
- 在 `onUnload` 钩子中显式清理：`allExternalWords = null`
- 实现 LRU 缓存限制 `getProfilesMap()` 的内存占用

**预期收益**
- 内存占用减少 60-70%
- 应用启动时间减少 30%
- 长期使用稳定性提升

**工作量**：7-9 小时

---

### 4. 缺少关键的错误处理和数据验证 🔴 高优先级

**问题描述**
- `databaseAdapter.js` 中 `query()` 和 `execute()` 方法缺少参数验证
- `learningCenter_v2.js` 中 `recordReviewOutcome()` 没有验证 `isCorrect` 参数类型
- 错误处理不一致：有些地方返回 `null`，有些返回 `[]`，有些抛出异常
- 没有对异常数据的恢复机制

**影响范围**
- 数据不一致：可能导致复习数据错误
- 应用崩溃：未捕获的异常导致页面白屏
- 用户数据丢失：没有事务回滚机制

**建议的解决方案**
- 实现统一的参数验证层：`validateWord()`, `validateReviewData()`
- 添加数据完整性检查：在保存前验证必填字段
- 实现事务回滚机制：在 `databaseAdapter.js` 中完善 `transaction()` 方法
- 添加数据修复工具：检测并修复不一致的数据

**预期收益**
- 数据一致性提升 95%+
- 应用崩溃率降低 80%
- 用户信任度提升

**工作量**：9-11 小时

## 🟠 第二阶段：重要改进（用户体验）

### 3. 重复的 JSON 序列化/反序列化 🟠 中优先级

**问题描述**
- `db_v2.js` 中 `parseWord()` 函数每次都调用 `parseJsonSafe()` 解析 examples/synonyms/antonyms
- `learningCenter_v2.js` 中 `safeRead()` 和 `safeWrite()` 每次都进行 JSON 序列化
- 没有缓存解析结果，导致重复计算
- 在大列表渲染时（1000+ 词），这会产生显著的 CPU 开销

**影响范围**
- CPU 使用率：高峰时 20-30% 用于 JSON 操作
- 列表滚动帧率：可能从 60fps 降至 30-40fps
- 搜索/筛选响应时间：增加 200-300ms

**建议的解决方案**
- 在 `parseWord()` 中使用 LRU 缓存存储已解析的对象
- 实现 `getWordByIdLight()` 和 `getWordByIdHeavy()` 分离
- 在列表渲染时只加载轻量级字段，详情页异步加载重量级字段
- 使用 `Object.freeze()` 缓存的对象防止意外修改

**预期收益**
- JSON 操作时间减少 70%
- 列表滚动帧率稳定在 55-60fps
- 搜索响应时间减少 200ms

**工作量**：5-7 小时

---

### 5. 复习算法的参数不一致 🟠 中优先级

**问题描述**
- `reviewAlgo.js` 中的 FSRS 算法参数与 `learningCenter_v2.js` 中的首日复习算法不同步
- `scheduleReviewState()` 计算的 `interval_days` 与实际使用的 `next_review_time` 可能不匹配
- 没有对算法参数的版本控制，升级算法时可能导致历史数据不兼容
- 不同页面使用不同的复习优先级计算方式

**影响范围**
- 复习计划不准确：用户可能在错误的时间被提醒
- 学习效率下降：算法参数不优化导致遗忘率高
- 用户困惑：不同页面显示的复习优先级不一致

**建议的解决方案**
- 统一算法参数：在 `algorithmConfig.js` 中定义版本号和参数
- 实现参数迁移机制：升级时自动转换旧数据
- 添加算法验证：在 `scheduleReviewState()` 中验证输出的合理性
- 创建算法测试套件：验证不同场景下的计算结果

**预期收益**
- 复习计划准确率提升 90%+
- 学习效率提升 15-20%
- 用户满意度提升

**工作量**：7-10 小时

---

### 6. 缺少虚拟滚动的页面 🟠 中优先级

**问题描述**
- `pages/stats/stats.vue` 和 `pages/word-detail/word-detail.vue` 中的列表没有虚拟滚动
- 当已掌握单词 > 1000 时，页面会卡顿

**影响范围**
- 页面加载时间：1000+ 词时可能 > 2 秒
- 滚动帧率：可能降至 20-30fps
- 内存占用：每个 DOM 节点 ~1KB，1000 词 = 1MB+

**建议的解决方案**
- 在 `pages/stats/stats.vue` 中为统计列表添加虚拟滚动
- 在 `pages/word-detail/word-detail.vue` 中为相关单词列表添加虚拟滚动
- 统一虚拟滚动配置：`itemHeight`, `bufferSize` 等

**预期收益**
- 页面加载时间减少 70%
- 滚动帧率稳定在 55-60fps
- 内存占用减少 80%

**工作量**：4-5 小时
